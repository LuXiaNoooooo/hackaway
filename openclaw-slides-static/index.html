Technical stack and system shape.

The product uses a shared home state model across frontend, simulation, control APIs, and OpenClaw orchestration. That means the interface, the automation layer, and the demo logic all describe the same home.

Frontend and simulation

  • React + Vite frontend for the 2D home console
  • Room-based floorplan editing and direct device interaction
  • Time, environment temperature, and robot motion simulation
  • Visual feedback for light mode, open state, temperature, and robot status

Control and orchestration

  • Capability-based APIs for lights, windows, doors, TV, climate, and robot
  • Room-aware selectors such as roomId, roomIds, ids, and all
  • Batch scene execution with one request and one final state
  • OpenClaw skill documents, API reference, and scenario playbooks

Core product capabilities.

This is not a flat device list. It is a home model that supports editing, live state visibility, semantic automation, and room-aware robot control.

Spatial editing

Users can define rooms, place devices, and understand the home layout at a glance.

Visible live state

Light modes, windows, AC temperature, time, and robot motion are visible on the map.

Semantic control

The API can target one room or the whole house instead of forcing one device at a time.

Room-aware robot cleaning

OpenClaw asks to clean the kitchen or the whole home; the system generates the route.

Batch scene execution

Multi-step routines are executed in order and return per-step results with final state.

Agent-ready documentation

OpenClaw receives explicit rules, selectors, endpoints, and working playbooks.

How OpenClaw connects.

The connection is explicit and inspectable. OpenClaw reads the home state, reasons over user intent, plans an ordered action chain, executes it with the batch API, and uses the returned final state to confirm what happened.

1. Read
OpenClaw calls GET /api/home/state to discover rooms, devices, and live conditions.
2. Reason
It translates user intent into room-aware outcomes like sleep mode, away mode, or TV ambience.
3. Plan
It builds ordered API actions, typically as one POST /api/actions/batch request.
4. Execute
The backend applies each action and returns step-level results plus the final state.
5. Verify
OpenClaw reports the outcome based on returned state instead of guessing or assuming.

Why this is stronger than HomeKit, Alexa, or Google Home for this use case.

Traditional smart home systems are convenient for simple device actions. This system is stronger when the goal is room-level reasoning, multi-step orchestration, and verifiable agent execution.

Traditional

Hackaway + OpenClaw

Traditional

Automation is often bounded by vendor integrations and limited scene builders.

Hackaway + OpenClaw

Traditional

Hackaway + OpenClaw

Demo plan.

The demo starts with the frontend itself, then moves through three natural-language scenarios that show why OpenClaw is useful in a spatial smart home system.

Frontend tour

Show the 2D home console.

Introduce rooms, devices, live visual state, temperature, time, and robot status.

Outcome: the audience sees a spatial operating layer, not a generic dashboard.

Scenario 1

"I just cooked and I want to rest."

Kitchen cleaning starts, lights turn off, bedroom switches to night mode, windows close, and climate moves to sleep comfort.

Outcome: room-aware reasoning across multiple device types.

Scenario 2

"I'm leaving home."

Powered devices shut down, windows and doors close, and whole-home cleaning begins.

Outcome: one coordinated away routine with clear safety and energy behavior.

Scenario 3

"I'm going to watch TV."

The robot stops, the rest of the house darkens, the TV room becomes cozy, and climate shifts for comfort.

Outcome: intent-based ambience, not just device-by-device toggling.
